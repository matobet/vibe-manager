# Phase 1 — UI Design Contract: Doorway Cards

> Visual and interaction contract for the manager doorway card on the root dashboard.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Medium note:** This is a terminal TUI (ratatui 0.29). All "pixels" are monospace
> character cells; all "fonts" are the user's terminal font; all spacing is in cells/rows.
> The web-oriented template sections below are adapted accordingly. Design direction:
> `docs/mockups/guild-halls.html` (strong direction, deviations noted), design contract
> prose: `docs/features/managing-managers.md` §2.1, theme source of truth:
> `src/theme/rpg.rs` + `src/theme/sprites.rs`.

## Spacing Scale

Terminal adaptation: units are **character cells (horizontal)** and **rows (vertical)**, not px. Scale is small-integer by medium necessity.

| Token | Value | Usage |
|-------|-------|-------|
| gutter | 2 cells | Sprite column → text column gap; gap between segments within a line (hearts → status, health % → outlier) |
| inline | 1 cell | Gap inside a segment (icon → text, `·` separators padded one cell each side) |
| badge-col | 4 cells | Right-aligned level badge column on line 1 (`Length(4)`) |
| sprite-col | 9 cells | Sprite column: 7-cell sprite left-aligned + 2-cell gutter (`Length(9)`) |
| card-height | 5 rows | 4 content rows + 1 blank spacer row (invariant — see DOOR-03) |
| health-bar | 10 cells | `▕` + 8 fill cells + `▏` |

Exceptions: none. No other ad-hoc padding; alignment comes from `Layout` splits, never manual space-padding of multi-byte strings (see Glyph Safety).

---

## Typography

Terminal adaptation: one font, one size (terminal-controlled). Hierarchy is expressed with exactly **2 weights** (regular, bold via `Modifier::BOLD`) and **brightness tiers** (white / theme color / gray), which serve the role of the type scale.

| Role | Cell Size | Weight | Color Tier |
|------|-----------|--------|------------|
| Identity (name, level badge) | 1 cell | bold | `COLOR_SECONDARY` when selected, `summary.color` (name) / `COLOR_SECONDARY` (badge) otherwise |
| Alert (outlier segment, `zZ`) | 1 cell | bold | `COLOR_DANGER` magenta |
| Data (health %, hearts, status) | 1 cell | regular | `COLOR_TEXT` / `mood_color()` / `overdue_color()` |
| Supporting (role title, `squad N`, hint, `(+N more)`) | 1 cell | regular | `COLOR_MUTED` gray |

Rule: at most one bold-magenta segment per card (the outlier alert) — bold is reserved for identity and the single worst-news item.

## Color

60/30/10 mapped to the existing palette (source: `src/theme/rpg.rs`, confirmed against mockup CSS vars):

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | terminal default bg + `COLOR_TEXT` white / `COLOR_MUTED` gray | Card body text, labels, supporting copy |
| Secondary (30%) | `COLOR_PRIMARY` cyan | Structural/positive: health bar fill, `✓` status, `★ all well` segment, header rules |
| Accent (10%) | `COLOR_SECONDARY` yellow | **Reserved for:** selected-card emphasis (sprite + name + badge), level badges, `⚠ you:` warning on line 2 when overdue |
| Destructive/Alert | `COLOR_DANGER` magenta | **Reserved for:** the named-outlier segment on line 3, `zZ` overdue suffix. Never red. |

Accent reserved for: selection emphasis, level badges, line-2 overdue warning — and nothing else on this surface. Magenta reserved for: outlier alert + `zZ` — at most two magenta segments per card, typically one.

**DECISION (resolves research Open Question 5):** Selected-card styling follows the **codebase convention (yellow `style_title()`)**, not the mockup's cyan frame + background fill. Rationale: IC cards already signal selection with yellow double-border; mixed selection colors across card types on one screen would be worse than mockup deviation. No background fill — the app never uses bg fills for cards, and terminal bg portability is poor. Source: `src/components/avatar.rs:42-57`, research recommendation.

## Component Contract: Doorway Card

New component `src/components/doorway_card.rs`, rendered by `AvatarGrid` for `report_type == Manager` as a full-width row; ICs keep `AvatarCard` untouched. No outer `Block`/border — the sprite frame is the visual anchor (mockup has none).

### Geometry

- Area: full dashboard width × **5 rows** (`DOORWAY_CARD_HEIGHT = 5`): rows 0–3 content, row 4 blank spacer.
- Horizontal split: `[Constraint::Length(9), Constraint::Min(0)]` → sprite column | text column.
- Sprite: `FaceSprite::from_summary` equivalent but **always via the non-overdue render path** — 4 lines, 7 cells, `Alignment::Left` in its column. The floating-`z` overdue sprite variant (up to 13 cells wide) is **forbidden** here: it would clip at 9 cells, and the mockup confirms overdue lives on line 2 as text (`zZ`) while the face keeps showing mood (Chris Wong, overdue, renders `◕‿◕`). Face expression = mood expression (`•_•` default when no mood), never `-_-` on this surface.
- Minimum supported width: 60 cells. Below that, lines clip at the right edge (ratatui default) — each line is composed so the lowest-priority info is rightmost (see truncation rules).

### Line-by-line contract (text column)

**Line 1 — Who.** Split `[Min(0), Length(4)]`; right cell is the badge, `Alignment::Right`.

```
JORDAN LEE — Eng Manager                                 M2
```

| Segment | Content | Style (unselected) | Style (selected) |
|---------|---------|--------------------|------------------|
| Name | `summary.name.to_uppercase()`, max 24 chars (`chars().take(24)`) | `summary.color` + bold | `style_title()` |
| Separator + title | ` — ` + profile `title` (verbatim, no abbreviation); **omit entire segment when title is absent/empty** | `style_muted()` | `style_muted()` |
| Badge | `summary.level` (e.g. `M2`) | `style_title()` | `style_title()` |

Note: `ReportSummary` lacks `title` today — adding it is in-scope (research DOOR-01 finding).

**Line 2 — Your relationship.**

```
♥♥♥♥♡  ✓ you: 3d            (healthy)
♥♥♥♡♡  ⚠ you: 8d  zZ        (overdue)
```

| Segment | Content | Style |
|---------|---------|-------|
| Hearts | `mood_gauge(recent_mood)`; `─────` when no mood | `mood_color(m)`; `style_muted()` when none |
| gap | 2 cells | — |
| Status | `overdue_icon(is_overdue)` + ` you: ` + `format_compact_age(days_since_meeting)` | `overdue_color(is_overdue)` (cyan ✓ / yellow ⚠) |
| Overdue suffix | `  zZ` — only when `is_overdue` | `style_danger()` |

`format_compact_age` (new helper in `theme/rpg.rs`): `None` → `never`, `< 14 days` → `{d}d`, `≥ 14` → `{d/7}w`. Used on lines 2 and 3. The mockup's `ago` suffix is dropped (mockup is internally inconsistent; compact form keeps width predictable).

**Line 3 — Their squad.** Three mutually exclusive shapes:

```
squad 4 ▕▓▓▓▓▓▓░░▏76%  ⚠ Sam T: mood ↘ · 6w (+2 more)   (outliers)
squad 5 ▕▓▓▓▓▓▓▓▓░▏90%  ★ all well · next: Ana P         (healthy)
squad 0 · no members yet                                  (no team)
```

| Segment | Content | Style |
|---------|---------|-------|
| Size | `squad {team_size} ` | `style_muted()` |
| Health bar | `health_bar(team_health_score, 8)` → `▕` + `▓`×filled + `░`×(8−filled) + `▏` | `style_header()` (cyan bold) |
| Percent | `{team_health_score}%` — immediately after bar, no gap | `COLOR_TEXT` regular |
| gap | 2 cells | — |
| Outlier (if any) | `⚠ {abbrev}: {label} · {age}` | `style_danger()` + bold |
| Overflow | ` (+{N-1} more)` — only when ≥ 2 outliers | `style_muted()` |
| Healthy (no outliers) | `★ all well · next: {abbrev}`; `★ all well` when no rotation candidate | `style_success()` |

- `abbrev` = `abbreviate_name()` (new util): `"Sam Taylor"` → `Sam T`, single names unchanged, first name + last-name initial.
- Outlier `label` priority (research A2, accepted): `mood_trend == Falling` → `mood ↘`; else `recent_mood ≤ 2` → `mood {m}`; else → `overdue`.
- `age` = `format_compact_age(outlier.days_since_meeting)`.
- Outliers ordered worst-first by urgency score; only the worst is named.

**DECISION (resolves research Open Question 4 — manager with no team):** when `team_metrics` is `None` **or** `team_size == 0`, line 3 is `squad 0 · no members yet`, entirely `style_muted()`. Never render a 0% health bar as if it were data. Both empty-`team/` and missing-`team/` shapes render identically.

**Line 4 — The door (DOOR-03).**

| State | Content | Style |
|-------|---------|-------|
| Selected | `▸ Space to visit squad` | `style_muted()` (mockup renders the hint dim even on the selected card — selection emphasis comes from sprite/name, not the hint) |
| Unselected | `Line::from("")` — **always emitted** | — |

Invariant: both states produce exactly 4 content lines + 1 spacer; selected and unselected buffers differ **only** in line 4's text column. Snapshot tests must assert this.

### Selected vs unselected summary

| Element | Selected | Unselected |
|---------|----------|------------|
| Sprite | `style_title()` (yellow bold) | `fg(summary.color)` |
| Name | `style_title()` | `fg(summary.color)` + bold |
| Title, `squad N`, `(+N more)` | muted (both states) | muted |
| Badge | `style_title()` (both states) | `style_title()` |
| Lines 2–3 data/alert segments | identical in both states | identical |
| Line 4 | hint (muted) | blank |

### Truncation / density rules

1. The card **never grows beyond 4 content lines** (spec §2.1 density rule). Multiple outliers collapse to `(+N more)`; nothing wraps.
2. Lines clip at the card's right edge; segments are ordered so clipping eats lowest-priority content first: line 1 loses title before badge is at risk (badge is in its own right column, never clipped by name), line 3 loses `(+N more)` then the outlier tail.
3. Name capped at 24 chars; titles render verbatim and clip naturally.
4. Never compose fixed-width padded strings around kaomoji or multi-byte glyphs; alignment is `Layout`'s job (research Pitfall 4 / mockup CSS comment).

### Optional in-scope copy (planner's call — research Open Question 3)

Dashboard header line, if the planner includes the `compute_extended_workspace_summary` wiring:

```
GUILD ROSTER · {direct_report_count} direct · {total − direct} in banner
```

Style: `COLOR_TEXT` bold, between the existing yellow `═` rules. If not included in Phase 1, this exact copy carries to Phase 2's breadcrumb work.

## Glyph Safety

All glyphs must be single-cell in common monospace fonts (project constraint). New glyphs this phase, with precedent:

| Glyph | Codepoint | Use | Safety |
|-------|-----------|-----|--------|
| `▕` | U+2595 | health bar left cap | Block Elements — same range as already-shipped `█░` (research A1; fallback: drop caps) |
| `▏` | U+258F | health bar right cap | same as above |
| `▓` | U+2593 | health bar fill | Block Elements, safe |
| `★` | U+2605 | all-well marker | already used in IC frames (`╔══★══╗`) |
| `▸` | U+25B8 | door hint marker | Geometric Shapes, single-width |
| `·` | U+00B7 | segment separator | Latin-1, safe |
| `—` | U+2014 | name—title separator | single-width em dash |
| `⚠` | U+26A0 **bare** | outlier/overdue warning | reuse existing `ICON_WARNING` exactly as shipped — **no U+FE0E selector** (consistency with current dashboard; the FE0E in the HTML mockup is a browser-only concern) |
| `♥ ♡ ✓ ↘ zZ` | — | existing | unchanged, via existing constants/helpers |

Forbidden: `⚠️` (emoji presentation), manual space-padding around `◦︵◦` outside `FaceSprite`, any new fullwidth glyphs.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (door hint) | `▸ Space to visit squad` — exact text, mandated by success criterion 3. Renders Phase 1; behavior lands Phase 2 (accepted one-phase mismatch; `input.rs` untouched this phase) |
| Empty state (no squad) | `squad 0 · no members yet` |
| Healthy state | `★ all well · next: {Name I}` / `★ all well` (no rotation candidate) |
| Alert state | `⚠ {Name I}: {mood ↘ \| mood {m} \| overdue} · {age}` + optional ` (+{N} more)` |
| Relationship status | `✓ you: {age}` / `⚠ you: {age}  zZ` |
| Never-met age | `never` (from `format_compact_age(None)`) |
| Error state | none new — existing status-bar pattern (`Error: {e}`, 3s auto-clear) covers load failures; a corrupt 2nd-level file must degrade to tolerant-load, not an error card |
| Destructive actions | none in this phase |

Voice: lowercase, terse, `·`-separated fragments — matches existing status bar and mockup throughout. No sentences, no punctuation terminators.
